Cassandra, and why writes are cheap but reads aren't

DistributedWide-columnEventually consistent12 min read

Cassandra is the database people reach for when writes have to never fall over — sensor data, event logs, anything append-heavy at massive scale. Understanding why it's shaped that way makes the "when to use it" question answer itself.

1No leader, no bottleneck

Every node in a Cassandra ring is a peer. There's no primary to route writes through and no single node that becomes the ceiling on throughput. A write can land on any node, which forwards it to the replicas actually responsible for that data's partition.

This is the opposite of a traditional primary-replica setup, where writes fight for one primary's attention. It's also why Cassandra scales writes almost linearly — add nodes, get more write capacity, close to proportionally.

2Why writes are fast

A write does two things, in order: append to a commit log on disk (sequential I/O, cheap), then insert into an in-memory structure called a memtable. The write is acknowledged as soon as both are done — no read-before-write, no lock contention, no random disk seeks.

Worth remembering

When the memtable fills up, it's flushed to disk as an immutable SSTable. Cassandra never edits data in place — it only ever appends. That single decision is responsible for most of its write performance.

3Why reads are slower

Because writes are append-only, a single row's data can be scattered across the memtable and several SSTables — a read has to check all of them and merge the results, picking the most recent version of each column along the way.

Cassandra mitigates this with bloom filters (to skip SSTables that definitely don't contain the key) and periodic compaction (merging SSTables back together) — but the fundamental shape is still "gather and reconcile," which a B-tree-indexed row store doesn't have to do.

4When to reach for it — and when not to

Good fit
  • Write-heavy, append-only data (logs, events, metrics)
  • Data naturally partitioned by a key you know in advance
  • Can tolerate eventual consistency
Poor fit
  • Complex queries across many columns you didn't model for
  • Strong consistency Requirements (financial transactions)
  • Small datasets where a single Postgres instance is simpler
