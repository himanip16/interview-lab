Interviewer
Let's say the payment succeeded, but our order service never received the event. What happens?
Candidate
I wouldn't retry the payment itself. I'd first separate payment state from event delivery.
Strength · ArchitectureCorrectly identifies that the external side effect and our system's awareness of it are two different failure surfaces \u2014 this is the crux of the whole problem.
Interviewer
Why?
Candidate
Because the external side effect may already have happened. My uncertainty is whether our system observed it \u2014 not whether the charge occurred.
Takeaway

"Separate payment state from event delivery; use the outbox pattern with idempotent consumers."

Interviewer
The client times out and retries the same payment request. What could go wrong?
Candidate
If we're not careful, we'd charge the customer twice \u2014 the retry looks like a brand new request unless we can recognize it as the same one.
Interviewer
The database write succeeds. The service crashes before publishing to Kafka. How do you maintain consistency between services?
Candidate
I'd use Kafka because Kafka guarantees message delivery and makes the system eventually consistent.
Gap · ConsistencyNames a technology instead of the mechanism. The actual gap \u2014 what happens between the DB commit and the publish \u2014 is never addressed here.
Interviewer
Right, but what happens in that specific gap \u2014 between the write succeeding and the event actually going out?
Candidate
Good point. I'd write the event to an outbox table in the same transaction as the payment record, then have a relay process publish from that table \u2014 so a crash before publishing just means the relay catches up later, nothing is lost.
